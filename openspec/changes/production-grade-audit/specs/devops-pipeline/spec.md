## ADDED Requirements

### Requirement: Multi-stage Dockerfile
The repository SHALL contain a `Dockerfile` implementing a multi-stage build: a `builder` stage that compiles TypeScript, and a `runner` stage based on `node:22-alpine` that contains only the compiled `dist/` output and production `node_modules`. The container SHALL run as a non-root `node` user.

#### Scenario: Docker image builds successfully
- **WHEN** `docker build -t bp:local .` runs from the repository root
- **THEN** the build completes without error and the resulting image is under 150MB

#### Scenario: Non-root user enforced
- **WHEN** `docker inspect bp:local` checks the user configuration
- **THEN** the image `User` field is `node`, not `root`

#### Scenario: Dev dependencies excluded from final image
- **WHEN** the runner stage is inspected
- **THEN** `devDependencies` packages (vitest, tsx, biome) are not present in the final image's `node_modules`

### Requirement: Semantic release automation
The release process SHALL be automated via `semantic-release` triggered on merge to `main`. Releases SHALL: bump the version in `package.json` following semver, generate/update `CHANGELOG.md`, create a GitHub Release with release notes, and publish to npm under `@agentic/bp`.

#### Scenario: Fix commit triggers patch release
- **WHEN** a commit with message `fix: ...` is merged to `main`
- **THEN** semantic-release creates a patch version bump (e.g., 1.0.0 → 1.0.1) and a GitHub Release

#### Scenario: Breaking change triggers major release
- **WHEN** a commit containing `BREAKING CHANGE:` in the footer is merged to `main`
- **THEN** semantic-release creates a major version bump (e.g., 1.0.1 → 2.0.0) and flags the release as breaking

#### Scenario: No releasable commits produce no release
- **WHEN** only `docs:` or `chore:` commits are merged to `main`
- **THEN** semantic-release skips the release without error

### Requirement: SAST workflow
A dedicated GitHub Actions workflow file `sast.yml` SHALL run CodeQL analysis on every PR targeting `main` and on every push to `main`. The workflow SHALL analyze the `javascript-typescript` language pack.

#### Scenario: CodeQL workflow triggers on PR
- **WHEN** a PR is opened targeting `main`
- **THEN** the CodeQL analysis workflow runs as a required status check

#### Scenario: SAST results appear in Security tab
- **WHEN** CodeQL analysis completes
- **THEN** any findings appear in the repository's GitHub Security > Code scanning alerts tab

### Requirement: Docker Scout vulnerability scan in CI
The CI pipeline SHALL run a Docker Scout image scan against every built Docker image, and SHALL fail the build if HIGH or CRITICAL vulnerabilities are found in OS packages or dependencies.

#### Scenario: Docker Scout scan blocks HIGH CVE
- **WHEN** the Docker image contains a package with a HIGH CVE
- **THEN** the Docker Scout CI step fails and the build is blocked

#### Scenario: Docker Scout produces SARIF report
- **WHEN** the Docker Scout scan completes
- **THEN** a SARIF report is uploaded to GitHub Code Scanning results

### Requirement: Release artifacts include SBOM
Every GitHub Release produced by semantic-release SHALL include a CycloneDX JSON SBOM file as a release asset, generated by `@cyclonedx/cyclonedx-npm` during the release workflow.

#### Scenario: SBOM asset present on release
- **WHEN** a GitHub Release is created
- **THEN** the release assets include a file named `sbom-<version>.json` in CycloneDX 1.4+ JSON format

### Requirement: CI environment matrix
The CI test workflow SHALL test against Node.js 20.x and 22.x (as currently configured) and additionally validate that the project builds and tests pass under Bun (stable) given the README's explicit Bun support claim.

#### Scenario: Tests pass under Bun
- **WHEN** the CI matrix runs tests using Bun runtime
- **THEN** all tests pass and no Bun-specific incompatibilities are reported

#### Scenario: Build artifact validated
- **WHEN** `npm run build` completes in CI
- **THEN** a validation step runs `node dist/cli/index.js --version` to confirm the compiled output is executable
