## Why

Project documentation needs a comprehensive regression review to align with codebase parity, resolve structural open-spec violations, and eliminate outdated or speculative content, ensuring a clean state for maintainers and users.

## What Changes

- Complete a full regression cycle over the documentation corpus (`docs/`, `specs/`, `README.md`, etc.).
- Generate structural and readability scorecards (`docs-readability-scorecard.md`, `docs-glossary-delta.md`).
- Formulate concrete documentation rewrite proposals (`docs-proposal/` or patches).
- Author the regression report (`docs-regression-report.md`) outlining critical/high/medium/low findings.

## Capabilities

### New Capabilities
- `docs-regression-cycle`: Run regression cycle on project documentation to ensure parity, readability, and compliance.

### Modified Capabilities

## Impact

No functional codebase impact. Enhances documentation parity and readability, reducing developer onboarding frictions and aligning docs with `bp` CLI implementation.
